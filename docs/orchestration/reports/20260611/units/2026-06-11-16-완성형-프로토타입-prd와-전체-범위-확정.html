2026-06-11-16 - 완성형 프로토타입 PRD와 전체 범위 확정

1. 현재 빌드 상태

Unity runtime 자체는 직전 Dev_Prototype_v0 상태를 유지한다. 이번 단위는 구현이 아니라 기준 재정의다. 이제 다음 구현 기준은 쌍검 1종과 기억 2종이 아니라 쌍검 + 대검 + 8기억 + 8잔향 + 4궁극 완성형 프로토타입이다.

2. 오늘 바뀐 것

  • LETHE_UNITY_COMPLETE_PROTOTYPE_PRD.md를 추가했다.
  • 디자인 문서 index에 완성형 PRD를 최신 실행 기준으로 등록했다.
  • NEXT_TASKS.md를 완성형 프로토타입 기준으로 교체했다.
  • CURRENT_TASK.md와 STATUS.md에서 다음 구현 방향을 data-driven core로 변경했다.
  • 대검, 8기억, 8잔향, 4궁극 잔향이 이제 프로토타입 검증 범위 안에 들어왔다.

3. 테스트 결과와 근거

  • 이번 단위는 문서/계획 단위라 Unity Play Mode 검증은 새로 실행하지 않았다.
  • npm.cmd run report: 통과, 16개 unit report 생성.
  • npm.cmd run report:check: 통과, 16개 unit heading 확인.
  • npm.cmd run report:orchestrator:unit:dry: fetch failed로 실패. Project Orchestrator intake endpoint가 현재 응답하지 않는 상태로 판단.

4. 결정한 것

  • 쌍검-only slice는 LETHE의 기준을 잡기에 부족하다.
  • 다음 기준은 Weapon_DualBlades, Weapon_Greatsword, 8 active memories, 8 echoes, 4 ultimate echoes다.
  • 다음 구현은 C1 Data-Driven Core부터 시작한다.
  • PrototypeGameManager에 모든 기억/잔향 분기가 쌓이는 구조는 폐기 대상이다.

5. 문제 또는 리스크

  • 범위가 넓어진 만큼 하드코딩으로 계속 밀면 다음 확장이 막힌다.
  • 대검과 전체 기억/잔향을 넣기 전에 데이터 계약이 흔들리면 다시 slice 수준으로 후퇴할 위험이 있다.
  • 아트 polish보다 각 효과의 역할 판독을 먼저 통과시켜야 한다.

6. GPT/Claude 인계 요약

사용자는 대검과 기존 기획의 기억/잔향을 모두 넣어 테스트하고 싶어 한다. 따라서 Codex는 완성형 PRD를 새 source of truth로 만들었다. 다음 Codex 작업은 C1 Data-Driven Core를 구현하고, 그 다음 C2 Weapon Pair, C3 Active Memories 8 L1 순서로 진행하면 된다.

7. 다음 Codex 작업

  • WeaponDefinition, MemoryDefinition, EchoDefinition, EchoLevelData, EchoSynergyDefinition 계약을 고정한다.
  • MemoryInventory, EchoInventory, ForgetService, ResonanceService, UltimateEchoService, DebugStateInjector를 분리한다.
  • 대검 runtime과 weapon switch debug를 만든다.
  • 8기억 level 1 동작을 debug panel에서 즉시 확인 가능하게 만든다.

8. 포트폴리오 메모

  • 문제: 너무 작은 slice로는 핵심 시스템의 기준을 판단할 수 없었다.
  • 방향: 완성형 prototype 범위를 먼저 확정하고 데이터 구조부터 넓힌다.
  • 행동: PRD, design index, orchestration state, next tasks를 한 번에 정리했다.
  • 결과: 다음 구현이 쌍검/2기억에 갇히지 않고 전체 게임 기준으로 진행될 수 있게 됐다.
