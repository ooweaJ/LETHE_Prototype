# 2026-06-02-14 - 프로젝트 목표 및 Claude 역할 정정 보고

## 1. 현재 빌드 상태

- LETHE HTML alpha는 v0.5 core-fun human-test ready 상태다.
- 현재 단계는 최종 게임 구현이 아니라 HTML 프로토타입으로 핵심 재미와 가능성을 검증하는 단계다.
- 충분한 재미와 가능성이 확인되면 그 결과를 근거로 Unity 구현 단계로 넘어간다.

## 2. 오늘 바뀐 것

- `AGENTS.md`, `CLAUDE.md`, `docs/CODEX_RUNBOOK.md`, `docs/CODEX_STATUS.md`에 HTML prototype -> Unity validation 목표를 명확히 적었다.
- Claude/GPT의 역할을 단순 검토자가 아니라 테스트 결과 해석, 기획 수정, 다음 구현 방향 결정 파트너로 정리했다.
- `docs/NEXT_TASKS.md`에 Unity 전환 판단 gate를 추가했다.

## 3. 테스트 결과와 근거

- 문서 변경 작업이라 게임 로직 테스트는 새로 필요하지 않다.
- 이 변경의 검증 기준은 운영 문서가 현재 목표를 일관되게 설명하는지다.
- 보고서 HTML은 Markdown 원본 기준으로 다시 생성한다.

## 4. 결정 사항

- Claude는 “리뷰만 하는 도구”가 아니다.
- AI 테스트와 사람 테스트 결과를 Claude/GPT에 보고하고, 그 결과를 바탕으로 기획을 수정한다.
- Codex는 수정된 기획을 작업 목록으로 바꾸고 구현, 테스트, 보고, 커밋을 맡는다.
- Unity 구현은 지금 당장 시작하지 않고, HTML 프로토타입에서 재미와 가능성이 확인된 뒤 판단한다.

## 5. 문제 또는 리스크

- 문서에 남아 있는 과거 “검토/리뷰” 표현은 당시 작업 기록일 수 있어 모두 지우지는 않는다.
- 앞으로 새 프롬프트와 보고서는 “검토”보다 “테스트 결과 기반 기획 수정”이라는 말을 우선한다.

## 6. GPT/Claude 인계 요약

- 다음 Claude/GPT 전달물은 단순 의견 요청이 아니라 플레이 결과 보고서여야 한다.
- 전달 내용에는 초반 재미, 성장 선택 반응, 망각 후회/짜증, 재시도 의향, Unity 전환 가능성 판단 질문을 포함한다.

## 7. 다음 Codex 작업

- 사람 테스트 결과가 모이면 `docs/review_prompts/`에 테스트 결과 기반 기획 수정 프롬프트를 작성한다.
- Claude/GPT 응답을 `docs/review_responses/`에 저장한다.
- 응답을 `docs/NEXT_TASKS.md`로 작업화한 뒤 다음 HTML 프로토타입 보완을 구현한다.

## 8. 포트폴리오 메모

- 문제: 자동화가 “검토 요청”처럼 보이면 실제 목표인 프로토타입 가능성 검증과 기획 수정 흐름이 흐려진다.
- 방향: HTML 프로토타입 검증 -> 테스트 결과 해석 -> 기획 수정 -> 다음 구현 -> Unity 전환 판단 구조로 정리한다.
- 실행: 프로젝트 규칙, Claude 역할, Runbook, 상태 문서, 다음 작업 문서에 목표와 역할을 반영했다.
- 결과: v0.5 이후 운영 기준이 “재미가 검증되면 Unity로 간다”는 프로토타입 목표에 맞춰졌다.
