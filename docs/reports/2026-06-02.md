- 앞으로 자동 루프는 구현 중간에 Claude 인증/권한/알림 문제가 터지는 방식이 아니라, 시작 전에 preflight로 위험을 확인하는 방식으로 운영한다.

## 2. 오늘 바뀐 것

- `scripts/autopilot_preflight.js`를 추가했다.
- `package.json`에 다음 명령을 추가했다:
  - `npm run autopilot:preflight`,
  - `npm run autopilot:preflight:local`,
  - `npm run autopilot:preflight:dry`.
- `scripts/check_local_pipeline.js`의 deep 검사에 preflight 문법 검사와 dry-run을 추가했다.
- `AGENTS.md`, `docs/CODEX_RUNBOOK.md`, `docs/ai/ai-collaboration.md`, `docs/CODEX_STATUS.md`에 preflight 규약을 반영했다.

## 3. 테스트 결과와 근거

- `node --check scripts/autopilot_preflight.js`: 통과.
- `npm run autopilot:preflight:dry`: 통과.
- `node scripts/autopilot_preflight.js --allow-dirty`: 15 pass, 2 warn, 0 fail.
- `npm run doctor:deep`: 43 pass, 0 warn, 0 fail.
- `npm run report`: `docs/reports/2026-06-02.html` 생성.
- `npm run report:discord:unit:dry`: `자동화 preflight 규약 보고` 섹션을 작업 단위 보고로 선택.
- `git diff --check`: 공백 오류 없음.

## 4. 결정 사항

- 무인 자동화 루프 전에는 반드시 preflight를 먼저 실행한다.
- `autopilot:preflight`는 최소 Claude 인증 프롬프트까지 보내므로, 신뢰된 로컬 터미널에서 사용한다.
- 외부 전송이 불가능한 Codex 세션에서는 `autopilot:preflight:local`을 사용하고, Claude 프롬프트/보고서 생성까지만 진행한다.
- preflight 실패는 참고사항이 아니라 blocker로 취급한다.

## 5. 문제 또는 리스크

- 실제 Claude 인증 확인은 외부 Claude 서비스 호출이므로, 현재 Codex 세션에서는 정책상 실행이 막힐 수 있다.
- git 작업 트리가 더러우면 preflight가 자동 루프를 막는다.
- Discord webhook이 없는 로컬은 알림 전송이 아니라 dry-run/문서 보고 중심으로 운영해야 한다.
- 현재 local preflight의 2개 warning은 의도된 검증 상태다:
  - 작업 중이라 git tree가 dirty였다.
  - live Claude auth는 외부 호출이므로 `autopilot:preflight:local` 경로에서 생략했다.

## 6. GPT/Claude 인계 요약

- Claude/GPT 피드백 루프는 계속 사용한다.
- 단, 자동 루프 시작 전에 인증/권한/알림/fallback 상태를 확인해 “중간에 안 될 수 있음”을 사전 문제로 바꾼다.

## 7. 다음 Codex 작업

- preflight 검증을 완료한다.
- v0.6 구현을 시작하기 전 `npm run autopilot:preflight:local` 또는 신뢰된 터미널의 `npm run autopilot:preflight`를 실행한다.
- preflight가 통과하면 구현 -> 테스트 -> 보고 -> Claude 피드백 -> 구현 루프를 반복한다.

## 8. 포트폴리오 메모

- 문제: 자동화를 표방하면서 Claude 인증/권한/알림 문제가 중간에 터지면 신뢰도가 떨어진다.
- 방향: 이미 아는 실패 조건을 자동 루프 시작 전에 검사하는 preflight gate로 만든다.
- 실행: preflight 스크립트, npm 명령, doctor deep 연결, 운영 규약 문서화를 추가했다.
- 결과: LETHE 자동화는 “될 수도 있다”가 아니라 “시작 전에 가능한지 검사하고, 실패하면 정확히 보고한다”는 구조로 정리됐다.
