# LETHE v0.8 전투/기억 코어 재기획 더블 체크 - 2026-06-02

## 목적

v0.7은 단순 밸런스 보정으로는 해결되지 않았다. 사용자 직접 플레이 결과, 현재 문제는 수치 조정이 아니라 전투/기억/시너지/테스트 모델을 크게 다시 세워야 하는 수준이다.

이번 프롬프트는 Claude와 Codex CLI 더블 체크용이다.

- Claude는 플레이 감정, 초반 리듬, 망각이 재미로 읽히는지 중심으로 판단한다.
- Codex CLI는 시스템 구조, 밸런스 모델, 구현 우선순위, 테스트 가능성 중심으로 판단한다.
- 현재 Codex는 두 답변의 공통점과 충돌점을 정리한 뒤 v0.8을 구현한다.

## 사용자 직접 피드백

- 밸런스가 전혀 맞지 않는다.
- 기억마다 성능 차이가 너무 크다.
- 기억 조합과 시너지 요소가 부족해서 빌드 만드는 재미가 약하다.
- HP가 1 이하로 떨어져도 죽지 않는 로직 때문에 AI 생존률이 높게 나온 것 같다.
- 실제 플레이에서는 절대 그렇게 오래 살 수 없다.
- 초반이 여전히 루즈하다.
- 첫 보스는 더 일찍 나와도 될 것 같다.
- 보스 간격은 선형으로 조금씩 늘어나는 방식도 괜찮아 보인다.
- 4분 첫 보스도 HTML 프로토타입 검증용으로는 길 수 있다.

## 확인된 기술/테스트 문제

- 현재 브라우저 게임에는 HP가 0 이하로 내려가도 `1`로 고정되는 생존 처리 로직이 있었다.
- 이 때문에 AI/QA가 실제 사망과 실패를 제대로 보지 못했다.
- 기존 AI 시뮬레이터는 실제 조작, 적 누적, 투사체 회피, 접촉 피해, 힐러 회복, 공간 압박을 충분히 반영하지 못했다.
- 기존 브라우저 QA는 boss -> forgetting -> deficit -> refill 흐름만 확인했고, 실제 전투 밸런스를 검증하지 않았다.

## 현재 v0.7 구조

- 총 런: 20분.
- 보스: 4 / 8 / 12 / 16 / 20분.
- 첫 보스 처치 후 기억 1개 상실.
- 이후 105초 동안 기억 2개로 결손 생존.
- 결손 후 새 기억 후보 3개 중 하나를 선택해 3기억으로 복귀.
- 기억은 총 6개, 시작 선택은 3개.
- 무기는 2개:
  - 절단쌍검: 빠른 안정형.
  - 장송대검: 느린 돌파형.
- v0.7에서 무기 기본 성능과 잃은 기억의 무기 잔향을 보강했지만 실제 플레이 체감은 실패했다.

## v0.8에서 검토할 큰 방향

### 1. 짧은 HTML 검증 런

HTML 프로토타입은 20분이 아니라 8-10분 안에 핵심 루프를 보여주는 방향이 나아 보인다.

후보:

- 첫 미니보스: 90초.
- 첫 망각: 90-120초.
- 다음 보스 간격: 2분 -> 2분 30초 -> 3분처럼 선형 증가.
- 전체 검증 런: 8-10분.

질문:

- 이 방향이 맞는가?
- 첫 망각은 정확히 몇 초가 적절한가?
- 보스는 미니보스/보스 구분이 필요한가?

### 2. 기억 성능 예산 재설계

현재 기억마다 실제 전투 기여 차이가 커서 잃는 결과가 재미보다 불합리로 느껴질 수 있다.

목표:

- 6개 기억 모두 단독으로 쓸 이유가 있어야 한다.
- 역할은 다르지만 전투 예산은 비슷해야 한다.
- 기억이 사라져도 "완전히 무너짐"이 아니라 "빌드가 다른 방향으로 변함"이 되어야 한다.

현재 역할 후보:

- 처형자의 섬광: 단일 폭딜 / 처형 / 보스 그로기.
- 굶주린 칼무리: 근접 광역 / 지속 피해 / 초반 잡몹 정리.
- 추적자의 맹세: 원거리 유도 / 안정딜 / 도망치는 적 대응.
- 파쇄의 파문: 광역 넉백 / 위기 탈출 / 군중 제어.
- 피의 반사: 흡혈 / 추가타 / 생존.
- 멈춘 초침: 둔화 / 쿨다운 / 위험 구간 제어.

질문:

- 이 역할 배분이 맞는가?
- 성능 예산을 어떻게 나눠야 기억 간 격차가 줄어드는가?
- 어떤 기억은 완전히 다른 역할로 바꿔야 하는가?

### 3. 태그/시너지 시스템

빌드 재미를 만들려면 기억 3개 선택이 조합으로 읽혀야 한다.

태그 후보:

- 폭딜
- 광역
- 지속딜
- 제어
- 생존
- 투사체
- 근접

시너지 후보:

- 광역 + 제어: 둔화/밀쳐낸 적에게 광역 피해 증가.
- 투사체 + 폭딜: 유도탄이 처형 표식 또는 폭발을 남김.
- 생존 + 근접: 근접 타격 시 보호막/흡혈.
- 지속딜 + 제어: 느린 적에게 출혈 tick 증가.
- 폭딜 + 생존: 보스에게 큰 피해를 넣으면 짧은 회복/보호막.

질문:

- HTML v0.8에서 태그/시너지 시스템을 넣어야 하는가?
- 넣는다면 최소 몇 개의 시너지가 필요하고, 어떤 조합을 먼저 구현해야 하는가?
- 시너지는 UI에 어떻게 보여줘야 하는가?

### 4. 망각 후 잔향 재설계

지금 잔향은 스탯 보상처럼 보일 위험이 크다.

목표:

- 잃은 기억의 태그가 무기나 남은 기억에 각인되어 빌드가 바뀌어야 한다.
- 잃어서 약해지는 것이 아니라, "한 축을 잃고 다른 축으로 변형"되어야 한다.

예시:

- 처형자의 섬광을 잃으면 무기 또는 남은 폭딜 기억에 처형 표식이 붙는다.
- 멈춘 초침을 잃으면 남은 광역/투사체 효과가 짧은 둔화장을 남긴다.
- 피의 반사를 잃으면 근접/지속딜 태그가 생존형으로 변형된다.

질문:

- 잔향을 태그 기반으로 바꾸는 게 맞는가?
- 기억을 잃은 뒤 어떤 수준의 보상이 적절한가?
- 잔향이 너무 보상처럼 느껴지지 않게 하려면 어떤 제약이 필요한가?

### 5. 테스트 모델 재설계

앞으로 `AI proxy pass`를 밸런스 통과로 말하면 안 된다.

필수 수정:

- HP 0 이하에서 실제 사망 처리.
- 사망률, 평균 사망 시간, 첫 사망 구간 기록.
- 처치 공백 시간.
- 화면 적 누적 수.
- 플레이어 HP 압박 시간.
- 결손 구간 중 처치/피해/사망 지표.
- 브라우저 combat QA에서 실제 전투를 일정 시간 자동 플레이하거나 위험도를 측정.

질문:

- v0.8 구현 전에 테스트 모델부터 고쳐야 하는가?
- 아니면 HP 사망 버그 + 런 구조 수정 + 기억 재설계를 먼저 하고 테스트를 같이 고쳐야 하는가?
- "사람에게 테스트 요청할 만하다"의 자동 기준은 무엇이어야 하는가?
