## 결론

- `BUILD_V09_RELEASE_FEEL_LOOP`

자동화 자체는 이미 준비 단계가 상당히 통과되어 있으므로, 지금 병목은 도구가 아니라 “한 사이클을 돌렸을 때 실제로 더 재미있는 빌드가 되는가”다. v0.9는 Gate C 브라우저 QA가 아니라, 출시작 감각을 내는 최소 런 구조를 먼저 만들어야 한다.

## v0.9 재미 목표

9분 HTML 프로토타입에서 플레이어는 90초 안에 “이번 런은 어떤 빌드인지” 읽어야 한다. 첫 압박 고점에서 특정 기억에 의존하게 되고, 그 기억을 잃었을 때 단순 약화가 아니라 “이 결손 상태로 버텨볼까?”라는 새 목표를 받아야 한다. 망각 후에는 남은 기억과 잔향이 조합을 바꿔, 실패가 아니라 변형된 런처럼 느껴져야 한다. 최종 목표는 “완성도 낮은 테스트 기능”이 아니라 “HTML인데도 한 판 더 해볼 만하다”는 반응이다.

## Codex 작업 단위

- [ ] **작업 1: 6개 기억 빌드 정체성 패스**
  - 각 기억에 1개 주 역할, 1개 태그, 20자 내외 전투 설명을 붙인다.
  - 3슬롯 조합에서 최소 3종의 빌드 방향이 즉시 보이게 만든다: 광역 제어형, 폭발 처치형, 생존/회복형.
  - 선택 UI와 전투 HUD에 “현재 빌드 이름 / 활성 시너지 / 의존 중인 기억”을 표시한다.
  - 목표: 90초 안에 AI 로그와 수동 테스트 모두에서 빌드 방향이 설명 가능해야 한다.

- [ ] **작업 2: 압박 beat + 망각 후 challenge 패스**
  - 9분 런을 `도입 -> 첫 빌드 고점 -> 압박 고점 -> 망각 -> 결손 도전 -> 변형 회복 -> 최종 압박`으로 나눈다.
  - 첫 망각 직후 60-90초짜리 post-loss challenge를 1개만 구현한다.
  - 가장 적합한 challenge는 **잃은 기억의 역할 없이 특정 압박을 넘기는 것**이다. 예: 광역 제어 기억을 잃으면 “몰려오는 적 안에서 Echo Zone 2회 통과”, 폭딜 기억을 잃으면 “엘리트 처치 대신 45초 생존”, 생존 기억을 잃으면 “피격 3회 이하로 버티기”.
  - 성공 보상은 새 기억 추가가 아니라 **잃은 태그의 잔향 변형**이어야 한다.

- [ ] **작업 3: 자동전투 수동 agency 최소 장치**
  - 가장 비용 대비 효과가 큰 장치는 **Echo Zone**이다.
  - 맵에 짧게 유지되는 구역을 만들고, 플레이어가 들어가면 특정 태그/잔향이 강화되지만 적 압박도 같이 올라간다.
  - 보스나 엘리트 tell은 Echo Zone과 연결한다. “도망만 가기”가 아니라 “위험하지만 들어가면 회복/증폭된다”는 판단을 만든다.
  - 새 무기, 새 지역, 새 기억 없이 위치 선택만으로 agency를 만든다.

## 밤샘 루프 기준

루프 1회는 다음 단위여야 한다.

1. `npm run autopilot:preflight` 또는 정책상 외부 호출이 막히면 `npm run autopilot:preflight:local`
2. 현재 `NEXT_TASKS`, `CODEX_STATUS`, 최신 loop log 확인
3. v0.9 작업 단위 중 하나만 구현
4. `npm run ai:test:quick`
5. 가능하면 `npm run ai:test`
6. 해당 기능 전용 브라우저/헤드리스 QA
7. `docs/CODEX_STATUS.md`, `docs/devlog/2026-06-02.md`, `docs/reports/2026-06-02.md` 갱신
8. 실패 원인을 다음 루프 입력으로 `docs/review_prompts/`에 남김

실패했을 때는 “테스트 실패”만 남기면 안 된다. 다음 루프가 바로 실행할 수 있도록 `실패 지표`, `의심 원인`, `다음 수정 후보 1개`, `절대 건드리지 말 범위`를 남겨야 한다.

사람이 깨어났을 때는 아래 문서만 보면 된다.

- `docs/CODEX_STATUS.md`
- `docs/NEXT_TASKS.md`
- 최신 `docs/loop_runs/*.md`
- 최신 `docs/reports/2026-06-02.md`

## 테스트 기준

- AI test:
  - 기존 `GO_CANDIDATE`만으로 통과시키지 않는다.
  - 추가로 `buildIdentitySeen`, `synergyActiveTime`, `pressureSpikeRecovery`, `lossChallengeStarted`, `lossChallengeCompleted`, `postLossRecovery`, `echoZoneUseCount` 같은 지표가 필요하다.
  - 핵심 실패 신호는 생존 실패보다 “망각 후 아무 변화 없이 DPS만 떨어짐”이다.

- Browser QA:
  - 90초 내 빌드 이름/태그/시너지 표시 확인.
  - 첫 망각 후 challenge UI가 뜨고 완료/실패 상태가 기록되는지 확인.
  - Echo Zone이 실제 화면에 보이고, 진입/이탈/효과 표시가 겹치지 않는지 확인.
  - 9분 QA 모드 또는 fast 모드에서 JSON payload에 v0.9 지표가 들어가는지 확인.

- 수동 1인 테스트:
  - 첫 90초 후 “내 빌드가 뭐였는지” 말할 수 있어야 한다.
  - 첫 망각 때 “아깝다” 또는 “이제 어떻게 버티지?” 반응이 나와야 한다.
  - post-loss challenge가 짜증나는 벌칙이 아니라 짧은 목표로 읽혀야 한다.
  - Echo Zone을 최소 2회 이상 의식적으로 밟거나 피해야 한다.
  - 테스트 후 “Unity로 만들면 더 살아날 것 같다”는 근거가 최소 1개 나와야 한다.

## 앞으로 해야 할 일

가장 먼저 Codex에게 `v0.9 Work Package 1`만 구현하게 해야 한다. 빌드 정체성이 90초 안에 읽히지 않으면 망각, Echo Zone, post-loss challenge 모두 감정적으로 작동하지 않는다.

그 다음 루프에서 post-loss challenge를 붙이고, 마지막에 Echo Zone을 붙인다. 순서는 `빌드 애착 -> 상실 -> 결손 상태의 선택`이어야 한다. 이 순서를 지켜야 망각이 짜증이 아니라 LETHE의 핵심 재미로 검증된다.
