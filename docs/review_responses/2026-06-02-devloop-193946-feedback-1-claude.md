## 결론

- **ITERATE_BEFORE_TEST**

## 이유

1. **범위 적합** — v0.9 WP2 Slice B(결손 압박) 구현이 `NEXT_TASKS.md`의 최우선 항목 범위 안에 정확히 들어왔다. 새 기억/슬롯/상점/지역 없이 스폰 파라미터만 조정한 점도 범위 제한 준수.

2. **AI 게이트 통과** — `GO_CANDIDATE`, Alpha Fun Score `0.8846`. `irritationRate: 0.0104`, `dullRate: 0`, `restartRate: 0.9`는 전부 양호. 망각이 짜증이 아닌 아쉬움으로 작동하고 있다는 봇 시그널은 확보됨.

3. **`earlyChoiceInterest: 0.6534`가 가장 약한 지표** — `earlyKillTempo: 0.9597`과 비교하면 전투 리듬은 살아있지만 선택의 재미가 부족하다. 사람 테스터가 망각까지 도달하려면 초반 선택 압박이 먼저 올라와야 한다. 지금 바로 사람 테스트로 넘어가기엔 이 수치가 낮다.

4. **`postLossChallengeScore: 0.6687`, `postLossChallengeContrast: 0.3134`** — Slice B 구조는 존재하지만 "결손 정비 → 결손 압박" 두 구간의 감각 차이가 충분히 선명하지 않다. 봇 기준이므로 사람이 체감할 때 더 약하게 느껴질 가능성이 있다.

5. **브라우저 QA 미확인** — `qa:postloss` Chrome/CDP timeout은 로컬 인프라 문제로 격리됐지만, Slice B 브라우저 실행이 아직 사람 손으로 검증되지 않았다. 커밋/푸시도 미완료 상태.

## 앞으로 해야 할 일

- [ ] **[WP2 마무리] `npm run qa:postloss` 로컬 재실행 → 통과 확인 후 커밋/푸시**  
  Chrome이 켜진 상태에서 재실행. 타임아웃 재발 시 `--timeout-ms 30000`으로 한 번만 더 시도. 통과하면 그대로 commit & push.

- [ ] **[WP3 첫 조각] 스테이지 진입 직전 "기억 집중" 2-선택지 표시**  
  2스테이지 이후 각 스테이지 시작 전, 현재 보유 기억 중 하나를 골라 해당 스테이지 동안 소폭 강화(예: 발동 확률 +15%)하는 단일 선택지를 띄운다. 새 기억/슬롯 추가 없이 기존 `memories` 배열과 스폰 타이밍만 사용. 목표: `earlyChoiceInterest` 지표 개선.

## 테스트 기준

- **AI 테스트 기준:**  
  `earlyChoiceInterest` > `0.72` 도달 여부. `postLossChallengeContrast` 현상 유지(`0.30` 이상).
